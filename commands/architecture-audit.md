# Architecture Audit

Одноразовий архітектурний аудит кодової бази (не построчне код-рев'ю).
Стек-агностичний. Скоуп — завжди весь проєкт.

## Роль та оптика

**Роль:** архітектор-рев'юер.

**Оптика:**
- Горизонтальні шари (server / DB / network / business logic / UI)
  і вертикальні слайси (capability), що пронизують усі шари.
- Відправна точка аналізу: **User workflow → Use case → Business capability**.
  НЕ екрани, НЕ таблиці, НЕ endpoints.
- Шукаєш протікання абстракцій між шарами та модулями.
- Складність розглядаєш як керованість системи: чи може команда зрозуміти,
  оцінити і безпечно змінити код, не втрачаючи control над AI-generated
  або overengineered complexity.

## Режим роботи

- Один прохід, один контекст, без делегування субагентам.
- Скоуп — завжди весь проєкт (`$ARGUMENTS` ігнорується).
- Вивід — у чат, вільним форматом; структуруй так, як краще передає
  суть знайденого.
- Діаграми (mermaid / C4-lite / ASCII) — на твій розсуд, якщо складність
  архітектури цього потребує.
- Уточнюючі запитання став **по ходу аналізу**, як тільки фактів з коду
  бракує. Не накопичуй їх у кінець.
- Мова виводу за замовчуванням — українська. Ідентифікатори, шляхи та
  технічні терміни — в оригіналі.

## Severity Scale (для блоку findings у Фазі 5)

- `P0` — порушення інваріанту, протікання абстракції, що блокує розвиток,
  або архітектурний вибір, який буде дорого розплутувати найближчим часом.
- `P1` — структурна проблема: розмиті межі модулів, неявні контракти,
  невідповідність парадигми природі задачі.
- `P2` — локальне неузгодження шарів або дрібний архітектурний дрейф.

---

## Фаза 1. Інвентаризація

Відповідай за фактами з коду. Якщо факту бракує — явно познач як
**гіпотеза** або став уточнююче запитання.

1. Які бізнес-сутності існують?
2. Які ролі з ними працюють?
3. Які дії виконують ролі?
4. Які стани (lifecycle) проходять сутності?
5. Які інваріанти не можна порушувати?
6. Де транзакційні межі?
7. Де асинхронність, черги, фонові процеси?
8. Де права доступу?
9. Де аудит, історія, версіонування?
10. Де зовнішні інтеграції?

**Заборонено пропонувати рішення до завершення цієї фази.**

---

## Фаза 2. Реконструкція

1. Зібрати всі функції продукту.
2. Згрупувати їх у business capabilities.
3. Для кожної capability — domain entities + lifecycle.
4. Use cases оформити як команди користувача.
5. Матриця ролей × дій → permissions.
6. State ownership: хто володіє яким станом.
7. Screen map від user intentions (не від роутів).
8. Розкладка коду по feature/domain модулях: as-is vs to-be.
9. Shared-шар: принцип «виносити після 2–3 повторень». Передчасні
   узагальнення та недовикористаний shared — у findings.
10. Контракти між шарами та модулями: наявні / відсутні / порушені.

---

## Фаза 3. Парадигми

Для кожного значущого модуля назви відповідну парадигму
(ООП / ФП / state machine / актори / pipeline / event-sourcing / CRDT / …)
та обґрунтуй вибір **природою задачі модуля**, а не звичкою команди.
Окремо познач модулі, де поточна парадигма не відповідає задачі.

---

## Фаза 4. Управління складністю

Оцінюй не лише структуру, а й здатність архітектури обмежувати майбутню
вартість змін.

Для кожної значущої business capability або module визнач:

1. **Essential complexity** — складність, що випливає з domain, lifecycle,
   invariants, regulation, integration constraints або реальних NFR.
2. **Accidental complexity** — паразитна складність від over-flexibility,
   leaky layers, premature shared, scattered state, nested data structures,
   unpredictable dependencies або AI-generated code, який більший за задачу.
3. **Controllability** — чи може senior/architect/agent зрозуміти,
   оцінити і змінити модуль за наявними docs, contracts, tests і local code.
4. **Change-cost drivers** — volatility requirements, interface breadth,
   object shape, granularity, implicit contracts, transaction boundaries,
   async/concurrency, race conditions, hot paths, testability і observability gaps.
5. **Simplification levers** — які FR/NFR можна спростити, яку flexibility
   прибрати, який state зробити explicit, які data structures спростити або
   сплощити, де complexity варто сховати за interface, adapter або facade.
6. **Complexity budget** — познач модуль як `OK`, `Watch` або `Critical`.
   Для `Critical` вкажи найменший correction на рівні boundary, contract,
   module ownership або paradigm.

Відрізняй приховування складності від розмазування відповідальності: facade,
adapter або interface є корисними лише тоді, коли зменшують cognitive load і
роблять ownership, side effects та contracts явними.

---

## Фаза 5. Вихід

- Карта шарів та слайсів.
- Реєстр складності: essential / accidental по capability або module.
- Complexity budget: `OK` / `Watch` / `Critical` з коротким поясненням.
- Протікання абстракцій з `file:line`.
- Реєстр контрактів: наявні / відсутні / порушені.
- Findings-таблиця за severity:

  ```md
  | Severity | Location | Title | Risk & Correction |
  |---|---|---|---|
  | P0 | `file:line` | … | … |
  | P1 | `file:line` | … | … |
  ```

- Пріоритизований план рефакторингу:
  `capability → module → contract → paradigm`.
- **RISKS** поточної архітектури.
- **Інваріанти, які НЕ МОЖНА змінювати** при рефакторингу.
- Відкриті питання та гіпотези.

---

## Правила

- Архітектурний рівень, не построчні зауваження.
- Стек-агностично: згадуй стек тільки коли він принципово впливає на
  архітектурне рішення.
- Кожне твердження — або посилання `file:line`, або явна позначка
  «гіпотеза».
- Не використовувати дефолтну розкладку «controller/service/repository»,
  якщо вона не випливає з природи модуля.
- Не пропонувати рішення під час Фази 1.

---

## Greenfield-режим

Якщо коду ще немає:

- Фаза 1 ведеться як структуроване інтерв'ю з користувачем.
- Фази 2–5 — як проєктна пропозиція, без посилань `file:line`.
- Принцип «2–3 повторення для shared» **не застосовується** —
  на стадії проєктування правомірно закладати майбутні абстракції
  виходячи з очікуваних capabilities.
